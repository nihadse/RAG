3. Constats et Observations

3.1. Méthodologie de détection des identifiants manquants

Au départ, plusieurs utilisateurs du fichier user présentaient un LOCAL_IDENTIFIER manquant.
Pour corriger cela, une vérification croisée a été effectuée avec la base de données RMPM disponible en interne.

Grâce à cette extraction RMPM, il a été possible :

de retrouver et renseigner les identifiants locaux manquants pour la majorité des utilisateurs,

de réduire considérablement les écarts entre les fichiers user et account,

bien qu’il subsiste quatre utilisateurs dont les identifiants locaux n’ont pas pu être retrouvés, même après recherche manuelle via la plateforme AMPM Web.



---

3.2. Liaison entre User File et Account File

Le champ LOCAL_IDENTIFIER apparaît comme la clé principale de correspondance entre les deux fichiers.
Cependant :

certains identifiants locaux existent dans le fichier user mais pas dans le fichier account, et inversement ;

certains identifiants locaux sont associés à plusieurs ENTITY_ID, rompant la logique de correspondance unique.


Exemples :

LOCAL_IDENTIFIER = 87970058839 apparaît dans les deux fichiers mais sous des ENTITY_ID différents.

LOCAL_IDENTIFIER = 87000001006 (dans le fichier comptes) ne correspond pas à 07800000134 (dans le fichier utilisateurs).



---

3.3. Cohérence logique et interprétation métier

Un même utilisateur peut être rattaché à plusieurs comptes, parfois sous des entités différentes.

Le fichier account contient des comptes sans utilisateur associé, pouvant correspondre à :

des comptes inactifs,

des comptes issus de migrations,

ou des créations automatiques sans rattachement utilisateur.



À l’inverse, certains utilisateurs du User File n’ont aucun compte associé.
Ces utilisateurs présentent néanmoins des connexions récentes, la dernière activité observée remontant à février 2025, soit la période de l’extraction.
Cela montre que leur profil est toujours actif dans Xiscash, même si aucun compte SWIFT n’est visible dans le fichier comptes.

Parallèlement, plusieurs comptes orphelins apparaissent dans le fichier comptes :
leurs dates de création remontent à 2022, ce qui suggère des ouvertures anciennes sans activité utilisateur identifiée depuis plusieurs années.

3.4. Incohérences d’identifiants

Plusieurs anomalies structurelles ont été relevées :

un même email utilisateur correspond à plusieurs USER_ID et LOCAL_IDENTIFIER,

ex. amar.abbes@axa.de est lié à plusieurs identifiants locaux et entités ;

allal@groupe-chilail.com apparaît sous plusieurs entités avec différents identifiants locaux.



Cela traduit des duplications de profils ou des créations multiples au sein de structures de groupe.


---

3.5. Chronologie de création des comptes

Dans le fichier account, le champ CREATE_DATE suit un schéma chronologique cohérent pour un même identifiant local :
chaque nouveau compte ajouté vient prolonger la relation existante.
Cette logique confirme une continuité d’activité client au niveau du LOCAL_IDENTIFIER plutôt qu’une création d’utilisateurs distincts.

4. Difficultés d’analyse

1. Filtrage par date de création :
Filtrer sur le CREATE_DATE dans le fichier comptes fausse les résultats.
De nombreux liens valides disparaissent, car certains utilisateurs existent avant la création de leurs premiers comptes.


2. Granularité de définition d’un utilisateur :
Pour un même LOCAL_IDENTIFIER, plusieurs USER_ID existent.
Il faut donc choisir si le comptage des utilisateurs doit se baser sur le USER_ID (vue technique) ou sur le LOCAL_IDENTIFIER (vue client logique).


3. Indicateurs d’activité limités :

NB_LOG_TOTAL représente les connexions cumulées depuis la création du compte, pas sur une période donnée.

NB_LOG_30DAYS ne donne qu’une vision ponctuelle.
Pour estimer une activité trimestrielle, il faudrait disposer de plusieurs extractions consécutives.





---

5. Structures et Modèles Détectés

L’analyse révèle les schémas suivants :

Un ENTITY_ID correspond généralement à un MOM unique et un LOCAL_IDENTIFIER unique, mais peut regrouper plusieurs USER_ID.

Un USER_ID peut être rattaché à plusieurs ENTITY_ID et LOCAL_IDENTIFIER, reflétant des droits multi-entités (cas des utilisateurs de groupes).

Les dates de création dans le fichier comptes progressent de manière logique, confirmant la continuité client au fil du temps.

1. Adopter le LOCAL_IDENTIFIER comme clé de correspondance principale entre les deux fichiers, tout en le validant dans le périmètre de chaque ENTITY_ID pour éviter les chevauchements entre entités.

3. Établir une cartographie complète des correspondances entre utilisateurs et comptes :

utilisateurs sans compte,

comptes orphelins,
